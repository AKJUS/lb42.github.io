How to tame the TEI tiger

Taming the TEI tiger

Lou Burnard

September 2018

Why is the TEI so daunting?

  • The Text Encoding Initiative provides a kind of encyclopaedia, defining and documenting hundreds of textual features you might want to distinguish in a marked-up text.
  • Because it seems hard for the beginner to find a way through its complexities, many assume they should use one of the predefined ‘minimal’ selections it proposes.
  • I disagree!
  • TEI is all about ‘taking back control’ ...

What would you like to tag here?

<pb n="4"/>A MONSEI- <lb/>GNEUR LE REVE- <lb/>rendissime Cardinal <lb/>du Bellay. <lb/>S <lb/> <c rend="lettrine">V</c>EV le Personnaige, <lb/>que tu ioues au Specta- <lb/>cle de toute l'Europe...

or about how they relate to data in the real world ...

Conclusion (tentative)

  • We need to decide what we are going to mark up in our texts before we start!
  • There is no single right answer to the question!
  • XML provides a well-established syntax in which to express our markup
  • The TEI provides a rich lexicon of concepts, expressible as XML tags

ODD is a good way of expressing our choice

What we need

Any markup system has two complementary needs …

A formal schema (using a formal language such as DTD, RELAX NG, W3C Schema, Schematron) to express rules about the markup such as :

  • which tags are available ?
  • in what contexts ?
  • with what attributes ?
  • with what values ?
  • subject to what constraints ?

and documentation explaining the principles underlying our editorial practice and choice of tags to users of the resulting system :

  • in different natural languages ;
  • in various document formats (PDF, MsWord, HTML, epub,...).

ODD : the basic notion (1)

One Document Does it all

A special XML vocabulary for defining....

  • XML element types independent of a particular schema language
  • public or private groups of such elements
  • patterns (macros)
  • datatypes
  • classes (and subclasses) of element
  • and schemas using the foregoing

And also for defining references which can pull into a schema

  • named components from the above list
  • objects from other namespaces

... which is also closely integrated with a set of traditional document markup elements

ODD : the basic notion (2)

A single ODD can combine multiple specifications for the same object

  • a ‘canonical’ version extracted from the Guidelines
  • one or more additional versions, partial or complete
  • these additional specifications can replace, partially modify, or delete existing ones
  • the status of each additional specification is made explicit by its mode attribute

A minimal element specification, for example

  • The <elementSpec> element has attributes module and ident identifying its module and its canonical name
  • The <desc> elements describe its intended usage in various languages
  • The <classes> element specifies to which TEI classes it belongs
  • The <content> element specifies its possible content (also expressed using ODD)
  • The <exemplum> element contains a usage example
  • Additional comments (<remarks>) and cross references (<listRef>) are provided

ODD is primarily a specification language...

The TEI scheme is specified using ODD. (In fact ODD was invented for that purpose)

The source code forTEI P5 (available from http://www.tei-c.org/release/xml/tei/odd/Source/) comprises :

  • 39 TEI-XML files, 25 of which contain a chapter of documentation, most of them discussing and specifying a set of related markup constructs, a ‘TEI Module’
  • 832 TEI-XML files, each of which specifies a single element, class, datatype or macro
    • 33 datatypes (teidata.xxxx) e.g. teidata.percentage
    • 118 model classes (model.xxxx) e.g. model.biblLike.xml
    • 73 attribute classes (att.xxxx) e.g. att.divLike.xml
    • 36 macros (macro.xxxx) e.g. macro.phraseSeq.xml
    • 567 element specifications, from ab.xml to zone.xml

... but ODD is also a customization language

The same system is used to specify your selection from the TEI cornucopia and to specify the cornucopia itself.

A customization ODD is expressed with reference to another ODD ; typically, but not necessarily, the current version of the Guidelines

  • by selecting particular modules
  • by choosing from the objects (elements, classes, datatypes, macros) provided by those modules
  • by removing or changing some of the attributes provided for those objects
  • by removing or changing some parts of the specifications (for example valList)
  • ... and possibly by adding entirely new specifications

How to make an ODD

An ODD is just a TEI-XML document like any other. You can edit and maintain it in oXygen or any other XML editor

There are however a few tools to simplify the initial creation of an ODD  :

  • http://www.tei-c.org/Roma provides a simple web interface; work is ongoing on a more modern version (romaJS) introductory tutorial
  • ODD By Example : generates an ODD from a set of marked up TEI files

The schemaSpec element

The <schemaSpec> element is used to ... specify a schema

  • attribute ident (mandatory) provides a name for the schema
  • attribute start indicates the root element(s) of the schema
  • attribute source indicates where specifications referenced by the spec are to be found (for example, a specific version of TEI P5)
  • attributes docLang and targetLang provide the ability to choose a different language for human-readable descriptions and for element or attribute names respectively (provided these have been translated in the source used)
<schemaSpec start="TEI" ident="testschema"  source="tei:1.5.0" docLang="ru"> <!-- declarations --> </schemaSpec>

Tricks

  • A single ODD document can contain more than one <schemaSpec> element, for example where these share a common set of <specGrp> elements
  • One ODD can serve as the base for another (‘oddChaining’)

The source attribute

The source attribute indicates a source for the declarations to be included in the schema  : by default this will be the most recent version of TEI P5.

or …

<schemaSpec ident="test2" prefix="tei_"  start="TEI" source="tei:1.5.0"> <!-- use TEI version 1.5.0 --> </schemaSpec>
<schemaSpec ident="test4"  source="myCompiled.odd"> <!-- use my previously compiled set of specifications --> </schemaSpec>

As they are continually evolving, it is useful to be able to link your ODD to a specific version of the TEI Guidelines !

Components of a schemaSpec

  • <elementSpec>, <classSpec>, <dataSpec>, <macroSpec> (etc.) : specifications for new objects
  • <elementRef>, <classRef>, <dataRef>, <macroRef>(etc.) : references to existing objects
  • <moduleRef> : a predefined and named collection of specifications

A moduleRef allows you to select the whole of predefined set of specifications or to choose only some of them

As elsewhere the source attribute indicates where the set is to be found : by default, in the most recent version of TEI P5

Some other components of an ODD specification

  • <dataSpec> allows you to define the datatype of an attribute value for example as date or integer. or as a string matching some regular expression
  • <valList> allows you to enumerate all the legal values for an attribute
  • <constraintSpec> allows you to express more complex constraints on element content (for example, if this attribute has this value, this other attribute is not permitted)
  • <model> allows you to specify default processing intentions (for example, this element should be formatted inline or as a block, this element presents alternative encodings of which only one should be used, etc.

For example

<schemaSpec start="TEI" ident="testschema">  <moduleRef key="core" include="p hi"/>  <elementSpec key="p" mode="delete"/> </schemaSpec>

The <p> element is suppressed

<schemaSpec start="TEI" ident="testschema">  <moduleRef key="core" include="p hi"/>  <elementSpec key="p" mode="change">   <desc>This element contains only text</desc>   <content>    <textNode/>   </content>  </elementSpec> </schemaSpec>

The content and description of the <p> element are changed as indicated, but its other properties (for example, its attributes) remain unchanged from the default

ODD processing

At present, the only complete ODD processor available is the suite of XSLT stylesheets provided and maintained by the TEI

  • All available specifications (*Spec et *Ref) are combined and canonicalised
  • Multiple declarations are resolved
  • The output is processed in a second step to generate a formal schema and documentation

The stylesheet odd2odd performs the first two steps and creates a compiled ODD which can be saved for reuse as the source of another ODD

An example

<body>  <head>A TEI customization for collaborative transcription</head>  <p>This customization proposes a very simple schema for use in a distributed editing    environment. Aside from a minimal TEI Header, it allows for the capture of the basic    organization of an archival document into divisions containing blocks of text. It    also allows for markup of page breaks, highlighted phrases, names, and illegible    passages. </p>  <schemaSpec ident="transMin"   start="TEI text div" docLang="fr">   <moduleRef key="tei"/>   <moduleRef key="header"    include="teiHeader fileDesc titleStmt publicationStmt sourceDesc"/>   <moduleRef key="textstructure"    include="TEI text body div"/>   <elementRef key="ab"/>   <elementRef key="pb"/>   <elementRef key="unclear"/>   <elementRef key="hi"/>   <elementRef key="name"/>   <elementRef key="title"/>   <classRef key="att.global.rendition"    except="rendition style"/>   <classSpec type="atts"    ident="att.declaring" mode="delete"/>   <classSpec type="atts"    ident="att.edition" mode="delete"/>   <classSpec type="atts"    ident="att.editLike" mode="delete"/>  </schemaSpec> </body>

This customization contains :

  • some documentation
  • a named <schemaSpec> element which specifies an identifier for the schema and its documentation language
  • a selection of elements to be included
  • a selection of attribute classes to be suppressed

Example contd

<!-- ... --><elementSpec ident="botName"  ns="http://myProject.com">  <desc xml:lang="ru">ботаническое название</desc>  <desc xml:lang="en">botanical name</desc>  <classes>   <memberOf key="model.phrase"/>   <memberOf key="att.global"/>  </classes>  <content>   <macroRef key="macro.paraContent"/>  </content> </elementSpec> <!-- ... -->
  • We add a non-TEI element, belonging to a different namespace; by making it a member of a TEI model class we control where it can be used within the TEI document
  • the new specification contains:
    • descriptive text in multiple languages
    • an indication of the TEI classes which the element will join
    • an indication of its content model

Example contd

<!-- ... --><elementSpec ident="hi" mode="change">  <attList>   <attDef ident="rend" mode="replace">    <datatype>     <dataRef key="teidata.enumerated"/>    </datatype>    <valList type="closed">     <valItem ident="underline"/>     <valItem ident="superscript"/>    </valList>   </attDef>  </attList> </elementSpec> <!-- ... -->
  • a part of the existing specification for the <hi> element is changed
  • specifically, the definition of its rend attribute is changed:
    • it gains a new datatype (an enumerated list)
    • its possible values are made explicit

Organization of an ODD

Aside from the <schemaSpec> element, an ODD document is structured in the same way as any other TEI document, using <div>, <list>, etc.

An ODD document can also use

  • special elements to organize groups of declarations within independently of a <schemaSpec> 
  • special elements to extract documentary parts of the specifications

Types of ODD (1)

each box here represents :

  • an ODD
  • the schema derived from that ODD
  • the set of documents considered valid by that schema

TEI All contains every element (etc) defined by the TEI.

Types of ODD (2)

  • a ‘TEI subset’ provides only a (possibly restricted) subset of TEI All
  • a ‘TEI extension’ provides some components which do not appear in TEI All

The $1000k question: is my TEI extension conformant?

The short answer

Yes, provided that...

  • (in the case of a subset) : your modifications generate a more focussed schema, better adapted to your project, and documentation more meaningful to your envisaged community;
  • your documents are also still valid according to TEI All and respect the semantics defined by the TEI conceptual model;
  • (for an extension) : non-TEI componments are explicitly signalled, for example by use of a different namespace, and documented in your ODD, for example by using TEI classes, the <equiv> ewlement etc.

So what does it mean to be TEI conformant ?

  • be honest : XML elements declared within the TEI namespace must respect the existing TEI definitions for those elements (e.g. <l>)
  • be explicit : an ODD is a very good way of keeping you honest. Producing one requires you to document and make evident all the changes you have made.

Validity of a document with respect to a TEI schema (TEI All, subset, or extension) is a good sign -- but it does not guarantee conformance

The requirement to ‘respect the TEI-defined semantics TEI’ implies a test that cannot be readily automated.

The limits of modification

  • Can you delete everything? you may not want <title> in your text, but it is mandatory in the header 
  • Can you add anything ? it may be convenient to add (e.g.) elements from the Dublin Core to your header, even though their semantics overlap with existing TEI elements
  • The purpose of these conformance rules is to make ‘blind interchange’ simpler; but they don't guarantee it.
  • Their goal is to allow a user to understand your encoding, but not necessarily to force them to follow your practice blindly
